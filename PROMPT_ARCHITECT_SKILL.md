# 🧬 PROMPT ARCHITECT v2.0
## Системный скилл контекстной инженерии

> **Философия:** Промпт — это не текст. Это **архитектура внимания модели**.
> Каждый токен либо фокусирует, либо размывает результат.

---

## ЧАСТЬ I: МЕНТАЛЬНАЯ МОДЕЛЬ

### Как думает промпт-инженер

```
НЕ ТАК: "Что написать модели?"
А ТАК:  "Какой контекст нужен модели, чтобы она сама пришла к нужному результату?"
```

**3 принципа:**
1. **Принцип Бритвы** — каждый токен должен работать. Промпт деградирует после ~3000 токенов (даже при окне 200K).
2. **Принцип Инверсии** — не описывай, что делать. Опиши мир, в котором правильный ответ — единственный логичный.
3. **Принцип Контекстного Окружения** — данные СВЕРХУ, инструкции СНИЗУ. Это даёт +30% к качеству (Anthropic, 2025).

### Иерархия влияния на результат

```
[1] Ясность и прямота          ████████████ (максимальный эффект)
[2] Примеры (Few-shot)         █████████░░░
[3] Структура (XML/MD теги)    ████████░░░░
[4] Chain-of-Thought           ███████░░░░░
[5] Роль / Персона             █████░░░░░░░ (минимальный эффект)
```

> Источник: Anthropic Prompting Best Practices, 2025. Большинство людей начинают с роли (5), хотя максимум отдачи — в ясности (1).

## ЧАСТЬ II: 7-КЛАСТЕРНЫЙ ФРЕЙМВОРК

### Порядок сборки промпта

```
┌─────────────────────────────────────┐

│  🎭 РОЛЬ                           │  ← Кластер 1: Ролевая модель
├─────────────────────────────────────┤
│  🌍 КОНТЕКСТ ЗАДАЧИ                │  ← Кластер 2: Контекст
├─────────────────────────────────────┤
│  🎯 ЗАДАЧА (глагол + артефакт)     │  ← Кластер 3: Объект
├─────────────────────────────────────┤
│  🎨 ФОРМАТ И СТИЛЬ                 │  ← Кластер 5: Стилистика

└─────────────────────────────────────┘
```

### Кластер 1: РОЛЕВАЯ МОДЕЛЬ 🎭

**Что:** Кто модель в этой задаче? Экспертность, тон, границы.

**Формула:**
```
Ты — [роль] с [N] годами опыта в [домен].
Твой стиль: [прилагательные через запятую].
Ты НЕ: [антироль — что модель НЕ должна делать].
```

**Усилители:**
| Приём | Пример | Когда |
|---|---|---|
| Мульти-роль | «Ты одновременно архитектор и security-аудитор» | Нужна критика с разных углов |
| Анти-роль | «Ты НЕ маркетолог — избегай хайпа» | Предотвращение дрифта стиля |
| Калибровка уверенности | «Если не уверен — скажи прямо» | Снижение галлюцинаций |

**Анти-паттерн:** Избыточная вежливость. Исследования 2025 года (arXiv 2510.04950) показали: прямые промпты дают 84.8% точности vs 80.8% у "вежливых". Не трать токены на «пожалуйста» и «будь добр».

---

### Кластер 2: КОНТЕКСТ 🌍

**Что:** Бэкграунд, аудитория, ограничения. Всё, что модель не знает, но должна.

**Формула:**
```xml
<context>
  <background>Мы строим [что] для [кого]. Сейчас на этапе [каком].</background>
  <audience>Читатель — [кто]. Знает [что]. НЕ знает [что].</audience>
  <constraints>
    - Бюджет: [X]
    - Срок: [Y]
    - Технический стек: [Z]
  </constraints>
</context>
```

**Правило шума:** Контекст полезен, пока он РЕЛЕВАНТЕН. Семантически похожий, но нерелевантный контекст — самый опасный тип шума. Модель путает его с инструкцией.

**Калибровка глубины:**

| Задача | Глубина контекста |
|---|---|
| Код: фикс бага | Минимум. Файл + ошибка + ожидание |
| Архитектурное решение | Максимум. Стек, ограничения, история решений |
| Текст: копирайт | Средне. Аудитория + тон + примеры конкурентов |

---

### Кластер 3: ОБЪЕКТ (ЗАДАЧА) 🎯

**Что:** Конкретное действие + конечный артефакт.

**Формула:**
```
ЗАДАЧА: [ГЛАГОЛ] + [АРТЕФАКТ]
```

**Словарь глаголов по типу задачи:**

| Категория | Сильные глаголы | Слабые (избегать) |
|---|---|---|
| Анализ | Декомпозируй, сравни, выяви | Посмотри, подумай |
| Создание | Сгенерируй, спроектируй, реализуй | Сделай, напиши |
| Трансформация | Рефакторни, конвертируй, оптимизируй | Измени, поправь |
| Оценка | Аудируй, ранжируй, верифицируй | Проверь, оцени |

**Декомпозиция больших задач (алгоритм):**

```
1. Опиши конечный артефакт (что должно получиться?)
2. Раздели на этапы, где каждый этап = отдельный промпт
3. Определи зависимости между этапами
4. Для каждого этапа: один глагол + один артефакт
5. Передавай выход этапа N как вход этапа N+1
```

```
Пример:
  Большая задача: "Напиши техническую документацию к API"
  ↓
  Этап 1: Извлеки все эндпоинты из кода → JSON-список
  Этап 2: Для каждого эндпоинта сгенерируй описание → Таблица
  Этап 3: Сгруппируй по доменам → Структура разделов
  Этап 4: Собери в единый документ → Markdown
```

### Кластер 4: ИНВЕНТАРЬ 📦

**Что:** Данные, примеры, ссылки — всё сырьё для работы модели.

**Размещение:** ВСЕГДА в начале промпта, до инструкций.

**Few-shot шаблон для кода:**
```xml
<examples>
  <example>
    <input>Функция сортировки массива</input>
    <output>
```python
def sort_array(arr: list[int]) -> list[int]:
    """Сортирует массив по возрастанию. O(n log n)."""
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return sort_array(left) + middle + sort_array(right)
```
    </output>
  </example>
</examples>
```

**Few-shot шаблон для текста:**
```xml
<examples>
  <example>
    <input>Тема: Микросервисы vs Монолит</input>
    <output>
## Микросервисы vs Монолит: когда что выбирать

**Монолит** — правильный выбор для команд до 10 человек и MVP.
Причина: единая кодовая база = быстрый деплой, простой дебаг.

**Микросервисы** — когда домены изолированы и команды автономны.
Причина: независимый деплой = скорость при масштабе.

Вердикт: начинай с монолита, разделяй по линиям боли.
    </output>
  </example>
</examples>
```

**Когда данные не помещаются в контекст:**

| Объём данных | Стратегия |
|---|---|
| < 3K токенов | Вставить в промпт целиком |
| 3K — 50K | Структурировать XML-тегами, вырезать нерелевантное |
| 50K — 200K | RAG: подавать только релевантные чанки |
| > 200K | Многоэтапная обработка: map-reduce через цепочку промптов |

### Кластер 5: СТИЛИСТИКА 🎨

**Что:** Формат, длина, тон, запреты.

**Формула:**
```xml
<format>
  Формат: [Markdown / JSON / код / таблица]
  Длина: [точное число слов/абзацев/строк]
  Тон: [технический / разговорный / академический]
  Язык: [RU/EN/смешанный с терминами на EN]
</format>

<constraints>
  НЕ используй: [список запрещённых слов/паттернов]
  ОБЯЗАТЕЛЬНО включи: [обязательные элементы]
</constraints>
```

**Токсичные слова (снижают конкретность ответа):**

| Слово-паразит | Замена |
|---|---|
| «Важно отметить, что…» | (удалить, начать с сути) |
| «В целом» | (конкретизировать) |
| «Различные» / «Разнообразные» | Назвать конкретно: «три типа: A, B, C» |
| «Достаточно» / «Довольно» | Число: «на 40% быстрее» |
| «И т.д.» / «и так далее» | Перечислить все или написать «и ещё N» |

### Кластер 6: ИТЕРАЦИЯ 🔄

**Что:** Как модель должна критиковать и улучшать свой первый ответ.

**Встроенная итерация (в одном промпте):**
```
Выполни задачу в 3 фазы:
1. ЧЕРНОВИК: Сгенерируй первый вариант.
2. КРИТИКА: Найди 3 слабых места в своём черновике.
3. ФИНАЛ: Исправь слабые места и выдай финальную версию.
```

**Алгоритм второго промпта (после получения черновика):**
```
Вот твой предыдущий ответ:
<previous_response>
[вставить ответ]
</previous_response>

Проблемы, которые я вижу:
1. [конкретная проблема]
2. [конкретная проблема]

Перепиши, устранив эти проблемы. Сохрани всё, что было хорошо.
НЕ начинай с нуля — эволюционируй существующий ответ.
```

**Паттерны рассуждений:**

| Паттерн | Триггер | Промпт |
|---|---|---|
| Chain-of-Thought | Математика, логика, код | «Рассуждай пошагово, покажи каждый шаг» |
| Tree-of-Thought | Стратегия, планирование, креатив | «Предложи 3 варианта → оцени каждый → выбери лучший» |
| Мета-когниция | Сложные / неоднозначные задачи | «Перед ответом: какие предположения ты делаешь? Какие из них могут быть неверны?» |

### Кластер 7: ВЕРИФИКАЦИЯ ✅

**Что:** Как проверить, что результат соответствует задаче.

**Встроенный чек-лист (добавлять в конец промпта):**
```
После генерации проверь свой ответ по критериям:
☐ Отвечает ли результат на исходную задачу? (точность)
☐ Нет ли выдуманных фактов/ссылок? (галлюцинации)
☐ Соблюдён ли заданный формат? (комплаенс)
☐ Можно ли удалить 30% текста без потери смысла? (лаконичность)
☐ Понятен ли результат целевой аудитории? (доступность)
Если хотя бы один пункт — нет, исправь перед выдачей.
```

**Метрика успешности промпта:**

```
Качество = (Релевантность × Точность × Полнота) / Длина_ответа

Где:
  Релевантность: 0-1 (ответ про то, что спрашивали?)
  Точность: 0-1 (факты верны?)
  Полнота: 0-1 (всё ли покрыто?)
  Длина_ответа: нормализованная (меньше = лучше при прочих равных)
```

**A/B тестирование промптов:**
```
1. Создай 2 варианта промпта (отличие в 1 переменной)
2. Прогони каждый 5 раз на одном и том же входе
3. Оцени по чек-листу выше
4. Вариант с большим средним — победитель
5. Зафиксируй: что именно изменило результат
```

### Сниппет 2: Промпт для код-ревью

```
Ты — Senior Software Engineer и Security Auditor (10 лет, Python/TypeScript).

<code>
{{КОД}}
</code>

ЗАДАЧА: Проведи код-ревью по 4 осям:

1. **Корректность** — баги, edge cases, race conditions
2. **Безопасность** — OWASP Top 10, инъекции, утечки данных
3. **Производительность** — O-нотация, лишние аллокации, N+1
4. **Читаемость** — именование, структура, избыточность

Формат для каждого замечания:
```
[SEVERITY: critical/warning/info]
Строка: N
Проблема: что не так
Исправление: конкретный код
```

Если код чистый по оси — напиши «✓ Чисто» и иди дальше. НЕ придумывай замечания ради объёма.
```

## ЧАСТЬ IV: ДИАГНОСТИКА ПРОБЛЕМ

### Промпт не работает? Чек-лист отладки

```
1. ЯСНОСТЬ
   □ Может ли человек без контекста выполнить эту задачу по твоему промпту?
   □ Есть ли двусмысленные слова? («это», «они», «тот код»)
   → Исправление: перепиши для пятиклассника

2. ДАННЫЕ
   □ Есть ли у модели ВСЕ данные для ответа?
   □ Данные размещены ДО инструкций?
   → Исправление: переместить данные наверх, добавить недостающее

3. ОБЪЁМ
   □ Промпт < 3000 токенов? (включая данные)
   □ Можно ли удалить 20% без потери смысла?
   → Исправление: вырезать «вежливость», дубли, очевидное

4. ПРИМЕРЫ
   □ Модель видит пример желаемого выхода?
   □ Пример покрывает edge case?
   → Исправление: добавить 2-3 few-shot примера

5. ФОРМАТ
   □ Указан точный формат выхода?
   □ Есть ограничения по длине?
   → Исправление: добавить <format> блок с конкретикой

6. ОПЕЧАТКИ
   □ Нет ли опечаток, лишних пробелов, сломанного форматирования?
   → Исправление: вычитать промпт как код (опечатки = баг)
```

## ЧАСТЬ V: АНТИПАТТЕРНЫ

| # | Антипаттерн | Почему плохо | Что делать |
|---|---|---|---|
| 1 | **Промпт-простыня** (>3K токенов) | Модель теряет фокус, качество падает | Разбить на цепочку коротких промптов |
| 2 | **Релевантный шум** | Похожий, но нерелевантный контекст путает модель сильнее, чем откровенно чужой | Безжалостно удалять «на всякий случай» |
| 3 | **Размытые глаголы** | «Подумай», «посмотри» — не дают модели цели | Заменить на конкретные: «декомпозируй», «сравни» |
| 4 | **Вежливость вместо ясности** | Тратит токены, не улучшает результат | Быть прямым: «Сделай X» вместо «Не мог бы ты, пожалуйста...» |
| 5 | **Инструкции до данных** | Модель забывает инструкции, дочитав данные | Данные → сверху. Инструкции → снизу |
| 6 | **Нет формата выхода** | Модель угадывает формат, результат непредсказуем | Всегда указывать формат + пример |
| 7 | **Промпт-дрифт** | Незадокументированные правки убивают качество | Версионировать промпты как код (git) |
| 8 | **CoT для простых задач** | Пустая трата токенов и времени | CoT только когда есть многошаговая логика |

## ЧАСТЬ VI: ЭВОЛЮЦИЯ — ОТ ПРОМПТОВ К КОНТЕКСТНОЙ ИНЖЕНЕРИИ

> «Промпт-инженерия — это оптимизация одной строки.
> Контекстная инженерия — это оптимизация всего, что видит модель.»
> — Anthropic, 2025

### Что входит в контекст кроме промпта:

```
┌─────────────────────────────────┐
│  System Instructions            │  ← Персона, правила, ограничения
├─────────────────────────────────┤
│  Tools / Functions              │  ← Описания доступных инструментов
├─────────────────────────────────┤
│  Retrieved Context (RAG)        │  ← Внешние документы, база знаний
├─────────────────────────────────┤
│  Conversation History           │  ← Предыдущие сообщения
├─────────────────────────────────┤
│  User Message (промпт)          │  ← То, что ты пишешь
└─────────────────────────────────┘
```

**Каждый слой — точка оптимизации.** Промпт-инженер работает только с нижним слоем. Контекстный инженер — со всеми пятью.

### Чек-лист контекстного инженера:

```
□ System prompt: минимальный, но полный набор правил
□ Tools: описания инструментов ясны и не конфликтуют
□ RAG: подаются только релевантные чанки, не весь документ
□ History: длинные диалоги сжимаются / суммаризируются
□ User message: конкретная задача + формат + верификация
```
